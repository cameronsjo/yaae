# Obsidian PDF Export Architecture Discoveries — Field Report

**Date:** 2026-03-26
**Type:** investigation
**Project:** YAAE (cameronsjo/yaae)

## Goal

Understand why watermarks, classification banners, headers, and footers were not rendering in Obsidian's PDF export despite correct CSS being generated by the plugin. Determine the actual architecture of Obsidian's PDF pipeline and what CSS mechanisms work within it.

## Root Cause

Three independent issues, each blocking a different PDF feature:

### 1. CSS Snippets Don't Reach PDF Export

**Finding:** Obsidian's `printToPDF()` pipeline does NOT include stylesheets from `.obsidian/snippets/`. Snippet CSS is loaded for the editor UI only.

**Proof:** Added `@media print { h1 { color: red !important; } }` to an enabled snippet — no effect in PDF. Added the same rule via a plugin-injected `<style>` element — H1 turned red in PDF.

**Impact:** The entire `@yaae/print-styles` package (watermarks, typography, tables, links, code blocks, page breaks, images) was a dead end as a snippet. All print CSS must be injected by the plugin at runtime.

### 2. `.print > div` DOM Doesn't Exist

**Finding:** The watermark CSS targeted `.pdf-watermark-heads-up .print > div::before`. The `.print` class and `> div` child structure don't exist in current Obsidian's PDF export DOM.

**Working approach:** The user's pre-YAAE watermark CSS (`print-draft-watermark.css`) applied `background-image` directly to the `cssclasses` element — no pseudo-elements, no `.print` container. Obsidian preserves `background-image` on the view container during PDF export.

**Fix:** Changed watermark from `::before` pseudo-element on `.print > div` to `background-image` directly on `.pdf-watermark-{level}`. Works, but only covers the content area (not full page).

### 3. `@page` Margin Boxes Need Chrome 131+

**Finding:** The PageChromeManager generates `@page { @top-center { content: "..." } }` rules. These are valid CSS Paged Media Level 3, but require Chrome 131+.

**Obsidian's Chrome:** Determined by the Electron installer version, NOT the app version. The app JS updates independently via Obsidian's self-update. Chrome version varies across machines:

| Machine | Obsidian | Electron | Chrome | Margin Boxes |
|---|---|---|---|---|
| Personal Mac | 1.7.5 | 32.2.2 | 128 | No |
| Work Mac | (same app) | (newer installer) | 132 | Yes |
| wdio E2E | 1.12.7 | (installer 1.5.8) | 120 | No |

**User agent check:** `navigator.userAgent` contains the Chrome version. Runtime detection is straightforward:
```typescript
const chromeVersion = parseInt(navigator.userAgent.match(/Chrome\/(\d+)/)?.[1] ?? '0', 10);
const supportsMarginBoxes = chromeVersion >= 131;
```

## What Works in Obsidian PDF Export

| Mechanism | Works? | Notes |
|---|---|---|
| Plugin `<style>` elements | Yes | Injected into `<head>`, included in `printToPDF()` |
| `@media print` (in plugin styles) | Yes | Triggers during export |
| `@page { margin }` | Yes | Sets page margins (Chrome 108+) |
| `@page { @top-center }` | Chrome 131+ only | Margin boxes for headers/footers |
| `background-image` on cssclass element | Yes | Tiling SVG watermarks work |
| `position: fixed` pseudo-elements | Yes | Repeats on every page (old approach) |
| CSS snippets (`.obsidian/snippets/`) | **No** | Not included in export pipeline |
| `.print > div` selector | **No** | DOM structure doesn't exist |
| `counter(page)` / `counter(pages)` | Chrome 131+ only | Part of `@page` margin box spec |

## Decisions Made

1. **Watermarks → background-image on cssclass element.** No pseudo-elements. Works on all Chrome versions. Trade-off: only covers content area, not full page (#25).

2. **Watermark presets → opacity-only ramp.** Same tile size/font/rotation across all levels. Whisper (3%) → heads-up (7%) → loud (14%) → screaming (25%). Previous approach varied tile density which felt like "more packed" rather than "louder."

3. **Always inject watermark CSS.** Previously conditional on custom text/non-default font. But the snippet fallback was broken, so defaults never rendered. Now the plugin always emits all watermark level rules.

4. **`showClassificationBanner` decoupled from PDF.** The setting now only controls the reading view post-processor banner. `PageChromeManager` always generates classification rules for PDF regardless of this setting.

5. **Use `cssclasses` (plural).** Singular `cssclass` silently fails in current Obsidian. The "Apply CSS classes" command merges with existing user classes instead of overwriting.

## Recommendations

- **Runtime Chrome version detection** should gate the PageChromeManager strategy. Use `@page` margin boxes on 131+, fall back to `position: fixed` pseudo-elements on older versions.
- **Eliminate the snippet package entirely** (#28). All print CSS should be plugin-injected. The snippet is a maintenance burden that provides zero value since Obsidian excludes it from PDF.
- **Eliminate `cssclasses` frontmatter** (#28). The plugin should inject CSS classes onto the DOM at runtime based on frontmatter, not persist them. Removes a manual step and frontmatter pollution.
- **Test PDF export on both Chrome <131 and 131+** to ensure the version detection fallback works across the install base.

## Key Takeaways

- Obsidian's PDF export has a narrow CSS surface: plugin `<style>` elements with `@media print` rules. Everything else (snippets, missing DOM, newer CSS features) fails silently
- Chrome version in Obsidian varies by installer date, not app version. Features that depend on Chrome 131+ will work for some users and silently fail for others
- `background-image` on the view container is the most reliable watermark approach — no pseudo-elements, no container selectors, works everywhere
- `position: fixed` pseudo-elements are the most reliable header/footer approach until Chrome 131+ is universal in Obsidian's install base
- The old working CSS from the user's vault (`print-draft-watermark.css`) was the Rosetta Stone — comparing its selectors to our broken ones immediately revealed the architectural mismatch
